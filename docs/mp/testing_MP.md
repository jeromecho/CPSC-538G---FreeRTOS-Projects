# MP - Testing

## Overview

The hardware-in-the-loop (HIL) testing framework described in the SRP documentation was employed and extended for SMP to generate reliable traces for SMP tests. Specifically, the following changes were made. First, the notion of a `core_id` was introduced into our captured traces. This allowed events to be associated with a particular core and plotted so that the specific cores an event was assocaited with could be visualized. Second, our regression testing test runner was fully automated. Instead of requiring the user to manually hold down the `BOOTSEL` button for the purposes of flashing a program, Picotool now automates the flashing of a program and the rebooting of a program. This now fully automates the running of all 69 regression tests.

Each test:

1. Defines a task set with specific parameters.
2. Creates all tasks (either on assigned cores in partitioned mode, or globally in global mode).
3. Allows the system to run for a (configurable) fixed duration.
4. Outputs collected traces.
5. Compares traces against expected behavior.

For partitioned scheduling, the first half of the tests were simply the tests defined for the EDF milestone, but run in parallell on both cores simultaneously. Since our approach manually assigns tasks to cores rather than using a partitioning strategy, verification of this was quite simple. Then, several tests were defined to verify that manual removal of tasks from cores and manual migration of tasks between cores worked as intended.

For global scheduling, initially we used the ReTIS Lab's RTSIM simulator to simulate global scheduling under EDF. However, due to minor real-time differences in the RTSIM simulator and our implementation, we realized midway into using RTSIM that RTSIM would not be a suitable tool for compariing our actual traces with the expected traces generated using RTSIM for all stress tests.

Based on this, we developed a verifier that analyzed the traces generated by a particular test for the following requirements: 1) Work Conservation, 2) EDF Guarantees (2 highest priority tasks always run), and 3) Execution Time (a task will always run for exactly its completion time). We developed unit tests to test our verifier and ran our verifier on tests that were already guaranteed to be correct by RTSIM. Then, we used our verifier on stress tests. The ability for the verifier to uncover small errors, such as off-by-one errors built our confidence in our implementation. We also sanity checked the plotted traces of tests after running the verifier. The verifier and tests can be seen here: https://github.com/jeromecho/CPSC-538G---FreeRTOS-Projects/tree/4-smp/_verifiers.

One fundamental limitation with using a verifier is that the correctness of the verifier is not guaranteed without extensive testing of the verifier itself. Meanwhile, using well-cited simulators like RTSIM gives us confidence that the expected traces generated by the simulator are correct. However, differences in the RTSIM simulator implementation (event-driven) and our implementation (tick hook based) and implementation details that prevented our generated traces from exactly equaling RTSIM's generated traces even with modifications to the RTSIM codebase made the usage of the verifier a better option, even with its limitations.

Finally, similar to our work for the EDF Documentation, we wrote a python script to validate our admission control functionality for global EDF scheduling. This script simulated the Bertogna and Cirinei's algorithm and produced the first admission control violation encountered. We did not provide detailed test descriptions for partitioned scheduling admission control because the tests replicated EDF admission control tests and used the exact same admission control algorithm. For more details on why particular task sets are admissible and some are not in the context of partitioned scheduling, please refer to EDF Documentation.
